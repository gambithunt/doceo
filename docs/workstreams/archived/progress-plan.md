# Progress Screen Plan

## Purpose

Redesign the student progress screen so it answers four questions quickly:

1. How am I doing overall?
2. What is improving?
3. What needs attention?
4. What should I do next?

The current screen shows basic counts and recent sessions, but it underuses the richer revision and learning-profile data already generated by the app.

This plan proposes a progress screen that is more useful, more specific, and more motivating without becoming noisy or gamified for its own sake.

## Design Alignment

Use the established Doceo design language in [`../design-langauge.md`](../design-langauge.md) as the main source of truth for layout, typography, spacing, dark-mode behavior, and component hierarchy.

Use [`design-color-01.md`](./design-color-01.md) as a secondary visual reference for:

- soft semantic grouping
- rounded floating cards
- subtle shadow depth
- colored surfaces that communicate meaning
- pill-based state indicators

Important: do not replace the Doceo dark-first system with the light reference directly. Instead, adapt the semantic color logic from `design-color-01.md` into Doceo's tokenized dark and light themes.

## Product Direction

The progress screen should feel less like a report and more like a learning cockpit.

It should prioritize:

- actionable insight over vanity metrics
- trend and durability over one-off scores
- next-step clarity over historical dumping
- encouragement grounded in real evidence

It should not prioritize:

- generic totals with no interpretation
- unreliable time-based claims
- decorative achievements without learning value
- dense tables that feel like admin software

### Existing Caveats

Do not headline these yet without more instrumentation:

- total study time
- lesson duration per session
- full session start/completion funnels based on analytics events

Reason:

- `LessonProgress.timeSpentMinutes` exists, but it is not a reliable session-level time model for this screen.
- several analytics event types are defined in types, but only a subset is currently emitted in the app.

## Screen Goals

The redesigned screen should:

- surface one clear overall health summary
- show what needs attention today
- celebrate meaningful improvement
- explain how the student learns best
- keep history available without making history the main event

## Proposed Screen Structure

## 1. Hero Summary Row

Replace the current top stats with four high-signal cards.

### Card 1: Memory Strength

Primary value:

- composite score from revision progress model

Meaning:

- overall durability of learning, not just recent confidence

Why it matters:

- this is a better headline metric than average confidence because it includes retention and revision consistency

Suggested copy:

- Title: `Memory strength`
- Support: `How well your learning is holding`

Visual direction:

- blue or teal informational tint
- stronger numeral treatment than current cards

### Card 2: Due Today

Primary value:

- number of revision topics due today
- secondary count for overdue topics if any

Meaning:

- gives the student an immediate next action

Suggested copy:

- Title: `Due today`
- Support: `Topics ready for a quick revisit`

Visual direction:

- yellow attention tint when there are due topics
- green success tint when clear

### Card 3: Consistency

Primary value:

- active learning days in the last 7 days

Secondary:

- mini 7-day bar strip

Meaning:

- turns progress into visible momentum

Suggested copy:

- Title: `Consistency`
- Support: `Days you showed up this week`

Visual direction:

- soft purple or blue

### Card 4: Confidence Calibration

Primary value:

- calibration status: aligned, overconfident, underconfident

Secondary:

- average self-confidence versus average correctness

Meaning:

- this is one of the most distinctive parts of the product
- it teaches metacognition, not just mastery

Suggested copy:

- Title: `Confidence coach`
- Support:
  - `You know more than you think`
  - `Confidence is running ahead of accuracy`
  - `Your confidence and answers are lining up`

Visual direction:

- blue for aligned
- yellow for caution
- pink for gentle encouragement if underconfident

## 2. Insight Row

Three cards that interpret the data for the student.

### Insight A: Breakthrough

Show:

- strongest recovered topic
- why it counts as a recovery

Example:

- `Transformations of Functions is in steady recovery`

Purpose:

- meaningful celebration
- not generic praise

Visual direction:

- green success tint

### Insight B: Watch Next

Show:

- most fragile topic
- short explanation of why it needs rescue

Example:

- `Fiscal Policy is still fragile`

Purpose:

- make the screen useful, not just pleasant

Visual direction:

- yellow attention tint

### Insight C: Learning Pattern

Show:

- one sentence summary derived from learner profile signals

Example:

- `You learn best when ideas are broken down step by step and tied to real examples.`

Purpose:

- turn abstract percentages into human guidance

Visual direction:

- purple or blue soft surface

## 3. Mastery Map

This should become the main body section.

Show:

- grouped by subject
- topic-level cells or rows
- latest confidence state for each completed topic
- active/incomplete topics shown differently from completed ones

Recommended format:

- desktop: grouped card grid or compact heatmap-like row sets
- mobile: accordion by subject with stacked topic rows

Status bands:

- strong
- steady
- fragile
- not enough attempts

Purpose:

- gives a true "where am I strong, where am I weak" view
- far more informative than a plain mastered/revisit list

Visual direction:

- use semantic tints from `design-color-01.md`
- avoid hard borders
- use pills and tinted cells instead of harsh color blocks

## 4. Revision Radar

A section dedicated to revision intelligence.

Show:

- topics due today
- overdue topics
- topics reviewed this week
- top misconception signal
- revision coverage in last 7 days

Optional compact cards:

- `3 due today`
- `1 overdue`
- `5 topics touched this week`
- `Main trap: confidence mismatch`

Purpose:

- connect revision system value directly to the student's daily behavior

## 5. Activity Timeline

Keep a history section, but make it richer than the current session list.

Combine:

- completed lessons
- revision attempts
- recovery moments
- reteach-heavy sessions

Each row should show:

- topic
- subject
- event type
- confidence or diagnosis badge
- relative date

Examples:

- `Completed • 78% confidence`
- `Revision • Confidence check`
- `Needs reteach • Explanation gap`

Purpose:

- history becomes meaningful, not just a log

## 6. Learning Style and Habits

Keep this section, but rewrite it as interpretation rather than raw telemetry.

Show:

- top 2 strongest support preferences
- question-asking habit
- reteach intensity
- number of subjects explored

Suggested presentation:

- one summary sentence
- two to four support chips
- one small secondary stat row

Example summary:

- `You usually do best when the lesson slows down, shows the steps, and gives a real example before practice.`

Supporting micro-stats:

- `24 questions asked`
- `8 reteach moments`
- `4 subjects explored`

Purpose:

- makes the profile feel alive and helpful

## 7. Session History

Move the current raw session history lower on the page.

Enhance each row with:

- confidence score if completed
- current stage if active
- reteach badge if present
- revision due badge if a completed lesson has become a revision topic

This section should be scannable, but secondary.

## What To Remove or Reduce

- generic `Active lessons` as a top-level headline stat
- raw mastered list without strength context
- raw revisit list based only on confidence under 70%
- learning-style percentages without interpretation
- session history occupying too much visual weight near the top

## Prioritized Metrics

## Ship First

- memory strength
- due today
- consistency this week
- confidence calibration
- strongest recovery
- most fragile topic
- learning pattern summary
- richer activity timeline

## Ship Next

- mastery map by subject and topic
- misconception radar
- stage/friction insights from lesson sessions
- subject exploration summary

## Later, After Better Instrumentation

- total study hours
- average session length
- exact completion rate funnels from analytics events
- speed-based metrics

## Visual System Notes

Adopt these rules from [`design-color-01.md`](./design-color-01.md):

- one floating card system
- semantic surface colors
- pill-based statuses
- soft shadow depth
- minimal border usage

Adapt them into Doceo's existing token system:

- dark mode remains the primary experience
- use tinted dark surfaces rather than pastel blocks in dark mode
- use warmer neutral surfaces in light mode
- keep the rounded card language
- maintain clear hierarchy and restrained copy

Recommended semantic mapping:

- green: recovery, completion, durable mastery
- blue: informational summaries, memory strength, neutral progress
- yellow: due soon, fragile topics, attention needed
- purple: learning pattern, study mode, reflective insights
- pink: playful celebration or gentle underconfidence feedback

## Mobile Behavior

The screen must remain strong on mobile.

Rules:

- hero cards become a stacked carousel or single-column stack
- insight cards stack vertically
- mastery map becomes accordions by subject
- timeline rows collapse into compact stacked rows
- no dense multi-column comparison blocks on narrow screens

## Suggested Information Architecture

Top to bottom:

1. Page intro
2. Hero summary row
3. Insight row
4. Mastery map
5. Revision radar
6. Learning style and habits
7. Activity timeline
8. Session history

## Implementation Notes

Prefer building a dedicated progress view model instead of placing all derivation logic directly in the Svelte component.

Suggested selector responsibilities:

- derive hero metrics
- derive due and overdue counts
- derive calibration summary
- derive recovery and fragility insights
- derive mastery map groups
- derive learning pattern summary
- derive unified activity timeline rows

This keeps the UI component clean and testable.

## Success Criteria

The redesign is successful if:

- a student can understand their learning health in under 10 seconds
- the screen clearly identifies a next action
- the screen celebrates real progress with evidence
- weak areas feel actionable, not discouraging
- the UI feels consistent with Doceo's design system and mobile standards
