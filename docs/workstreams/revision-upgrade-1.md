# Revision Upgrade 1

## Purpose

This workstream addresses a set of interconnected bugs and UX gaps found during live testing of the active revision session flow. It covers visual fixes, a rethought progression model, inline scaffolding for stuck students, topic history improvements, and the subject metadata bug.

The goal is to make revision feel like a real learning loop — not a single-question dead end. Students should move forward, get meaningful help at each step, understand what they got right and wrong, and be able to repeat the process.

This document is the working guide. All tasks are listed at the bottom.

---

## Context: What Is Currently Broken

### 1. Textarea border clips left (Image 3)

**What the student sees**: The answer textarea's teal border disappears off the left edge of the screen on the active revision view.

**Root cause**: A CSS containment or overflow issue on the `.revision-answer-field` wrapper or its parent flex/grid container. The border is rendering outside the clipping boundary of the parent.

**Fix**: Audit padding/margin on `.revision-answer-field` and ensure the container does not clip overflow. Check `min-width: 0` on flex children is set so the textarea doesn't expand beyond bounds.

### 2. Question card typography hierarchy (Image 4)

**What the student sees**:
- The RECALL pill has no breathing room below it
- Below the pill sits an `<h3>` with the topic title — this is the same title that already appears prominently in the hero section above the card. It creates a confusing double-heading.
- The yellow synthetic warning ("You haven't taken a lesson on this topic yet") sits between the heading and the question, interrupting the reading flow at the worst moment
- No bottom spacing on the RECALL pill

**Root cause**: The `session-question-card` renders an `<h3>` with `displayTopic?.topicTitle` even though the hero section above already shows that title at a larger size. The warning is placed before the question text instead of after.

**Fix**:

- Move the `.synthetic-topic-note` warning to below the question text (after the `<p>{currentQuestion.prompt}</p>`)
- Add `margin-bottom: 0.5rem` (or equivalent token) to the RECALL pill element

### 3. Nudge / Hint / I'm Stuck do not behave as expected (Image 5)

#### Current behaviour

**Nudge and Hint**: Both call `requestRevisionNudge()` / `requestRevisionHint()` in `app-state.ts`. These functions correctly set `revisionSession.currentHelp` with intervention content generated by `getRequestedIntervention()`. The `feedbackMarkdown()` function checks `currentHelp` first and renders it in a `feedback-card` article — but this card appears below the submit actions and is not visually anchored to the question. The student receives no visual signal that something appeared.

**I'm Stuck**: Calls `markRevisionStuck()` which sets `revisionSession.status` to `'escalated_to_lesson'`. This immediately makes `isSessionActive = false`, which collapses the entire question UI and shows the session-complete/escalation screen. The student loses their context and ends up in what feels like an error state.

#### Planned behaviour

**Nudge**: Show a short orienting prompt inline — between the question and the textarea, in a visually distinct but subtle container. Something like "Start with the core rule, then give one concrete example." The student can read it and immediately write their answer. No page transition, no state change beyond `currentHelp`.

**Hint**: Show a more structured scaffold inline — the hint content from `buildInterventionContent('hint', topic)`. Slightly more prominent than a nudge. Should appear with a brief transition so the student sees it arrive.

**I'm Stuck**:
1. Show a structured worked-step response inline — do NOT immediately escalate to `'escalated_to_lesson'`
2. The worked step appears in the inline help area with a label like "Worked steps"
3. Below the worked steps, offer a secondary button: "Take me back to the lesson for a full walkthrough" — this is the explicit escalation action
4. Only set `status: 'escalated_to_lesson'` when the student clicks that secondary button, not when they click "I'm stuck"

This way the student stays in the revision flow, reads the worked steps, and can attempt the answer again with scaffolding visible.

#### Implementation notes

- Add a `currentHelpVisible` reactive variable that becomes true when `revisionSession.currentHelp` is set
- Render an `inline-help` panel inside the `session-question-card`, between the question text and the textarea
- The panel should have three visual tiers: `nudge` (subtle, single sentence), `hint` (structured, slightly elevated), `worked_step` / `mini_reteach` (full worked example, clearly labelled)
- Animate the panel appearing with a `fly` transition from above
- `markRevisionStuck()` in `app-state.ts` should be changed: set `currentHelp` to the worked step content, set `currentInterventionLevel: 'worked_step'`, but do NOT set `status: 'escalated_to_lesson'`
- Add a new action `escalateToLesson()` that explicitly sets `status: 'escalated_to_lesson'` — called from the "Take me back to the lesson" button in the UI

### 4. Subject pill shows wrong subject (Image 6)

**What the student sees**: The header and pill row show "Mathematics" for a topic ("Understanding climate change impacts") that is clearly Geography.

**Root cause**: `displayTopic.subject` and `sessionRootTopic.subject` come from `RevisionTopic.subject`. For synthetic topics created via `buildSyntheticTopic(plan, topicTitle, now)` in `src/lib/revision/plans.ts`, the subject is taken directly from `plan.subjectName`. If a Geography topic was added to a Mathematics revision plan (e.g., TEST1), the synthetic topic records `subject: 'Mathematics'` permanently.

**Fix**:
- In `buildSyntheticTopic`, if the topic title contains strong curriculum keywords (e.g., "climate", "geography", "biome"), do not blindly inherit the plan subject — instead fall back to a `subjectHint` parameter that the caller can provide
- More robustly: in `buildPlanTopicSet` (or wherever synthetic topics are created), pass through the correct subject name if it can be inferred from the curriculum data
- Add a one-time migration in `normalizeAppState` (in `app-state.ts`) that checks for synthetic revision topics whose `subject` doesn't match their `curriculumReference` and corrects them
- Display guard: if `displayTopic.subject` and the topic title's curriculum reference disagree, prefer the curriculum reference for display

**Open question for user**: Was "Understanding climate change impacts" added manually under the wrong subject, or via the revision plan builder? Knowing this will determine whether the fix is in `buildSyntheticTopic` or in the plan builder UI (which should validate subject assignment before creating topics).

### 5. System only asks one question and does not advance (core flow bug)

**Root cause**: `applyRevisionTurn` in `engine.ts` only advances `questionIndex` when `result.sessionDecision === 'continue'`, which requires `scores.correctness >= 0.6` OR `(scores.reasoning >= 0.6 AND scores.completeness >= 0.55)`. For a student with a weak or short answer, `sessionDecision` becomes `'reschedule'`, which keeps `questionIndex` unchanged and sets `shouldRepeat = true`. The session status stays `'active'` but the student is shown the exact same question again with no indication of why, no way to skip, and no feedback visible in the right place.

With a `deep_revision` session (2 questions), if the student answers poorly on Q1, they loop on Q1 indefinitely. If they answer Q1 well but it's also the last question, the session immediately completes.

**What the student needs**:
1. After submitting any answer: see immediate, clearly-placed feedback — what their answer showed, what was missing
2. A "Next question →" button visible after feedback, regardless of score — the student can choose to move forward even if Doceo recommends repeating
3. A "Try again" path if they want to attempt the same question after seeing feedback
4. On the last question: a "Finish round" button

**Plan**:

After `submitRecall` resolves:
- Enter a `post-answer` state that shows:
  - A **diagnosis card** inline — the `diagnosis.summary` text from `lastTurnResult`, the calibration note, and the score (e.g., "3/5 correct · Confidence aligned")
  - If `currentHelp` was active: fold it into the feedback so the student sees help + result together
- Below the diagnosis card, show two buttons:
  - Primary: **"Next question →"** (if there are remaining questions) or **"Finish round"** (if last question)
  - Secondary: **"Try again"** (re-attempt the same question; clears the textarea, keeps intervention visible)
- `nextQuestion` in the engine already populates `lastTurnResult.nextQuestion` when the session is repeating — use this to tell the student "You'll see this again"

**Engine changes needed**:
- `applyRevisionTurn` should also set a `sessionDecision: 'force_advance'` path when the student explicitly chooses to skip — add a `forceAdvance` flag to the function signature
- Keep the existing auto-advance logic; `force_advance` just overrides it

### 7. Topic history — explain what the student got right and wrong (Image 7)

**What the student sees**: The topic history panel shows previous attempts with a diagnosis label ("Explanation gap"), a date, accuracy/confidence stats, and a summary sentence. But there is no breakdown of what was good in the answer, what was missing, or what to focus on next time. Learning cannot happen from just a label.

**Plan**:

Add an expandable "Review" section to each history entry that shows:

- **What you showed**: a sentence pulled from the `diagnosis.type` (e.g. "You had the right core idea but the logic chain was incomplete")
- **What was missing**: the `intervention.content` from the stored attempt result (the nudge/hint/worked-step content Doceo would have given — the "correct answer scaffold")
- **Model answer**: the output of `buildInterventionContent('worked_step', topic)` — deterministic, no AI call needed for the first version
- **What to do next**: the `sessionDecision` turned into plain language ("Doceo scheduled this for review tomorrow")

This is purely deterministic in v1. A v2 option is to send the student's answer + topic + diagnosis to the AI API for a personalised breakdown — this should be an explicit "Get AI feedback" button, not automatic.

**Implementation notes**:
- The stored `RevisionAttempt` already has `result: RevisionTurnResult` which contains `scores`, `diagnosis`, `intervention`, `sessionDecision`, and `topicUpdate`
- The history model in `deriveRevisionTopicHistoryModel` needs to pass through the `intervention.content` and `scores` from each attempt
- Add a `showReview` boolean toggle per history entry (local component state, not persisted)
- The expanded review section should feel like a mini debrief, not a data dump

### 8. No option to move to the next question

This is covered in §5 above (the "Next question →" button). Calling it out separately because it is the single most disorienting gap — after submitting, the student has no clear call to action.

**The feedback-then-advance pattern**:

```
[Student submits answer]
  → Show diagnosis card (inline, below textarea)
  → Show: "Next question →" | "Try again"
  → If last question: "Finish round"
[Student clicks "Next question →"]
  → Advance questionIndex (force_advance if needed)
  → Clear textarea, clear feedback, animate next question in
[Student clicks "Try again"]
  → Clear textarea
  → Keep inline help visible if it was showing
  → Re-focus textarea
```

---

### 9. Revision as progression — the full model

The student should feel that revision is a winnable loop, not a treadmill. The complete intended experience:

1. **Start**: session begins, first question appears
2. **Answer**: student writes their answer, picks a confidence level
3. **Check**: Doceo evaluates the answer, shows a diagnosis inline
4. **If weak**: Nudge / Hint / Worked steps are available; student can re-attempt
5. **After help**: "Show me the model answer" becomes available (gated behind at least one failed attempt or explicit hint use)
6. **Advance**: student moves to next question voluntarily
7. **Complete**: session ends with a summary — what was secured, what still needs work
8. **Restart**: "Practise this topic again" button starts a fresh session

**Model answer / "Show me the answer"**:
- Available after: student has submitted at least once AND either scored below threshold OR has used a nudge/hint
- Shows `buildInterventionContent('worked_step', topic)` content in a clearly-labelled panel with a label like "Model answer"
- Should not short-circuit the learning loop — shown after the student has genuinely tried

**Restart revision**:
- "Practise again" button in the post-session summary
- Calls `review(sessionRootTopic, 'deep_revision')` (the existing `review()` function)
- Also available in a lighter version: "Quick-fire this topic" for a 1-question spot check

## Open Questions

These need answers before or during implementation:

1. **Subject bug source**: Was "Understanding climate change impacts" added manually under the wrong subject, or was it generated by the revision plan builder with a wrong subject? This determines whether the fix is in `buildSyntheticTopic`, in the plan builder UI, or in both.

2. **"I'm stuck" worked-step tone**: Should the inline worked steps be instructional ("Here is how to approach it: 1. Define, 2. Explain, 3. Example") or Socratic ("What part is confusing you?"). The current `buildInterventionContent('worked_step')` is instructional. Changing the tone is a copy change, not a code change.

3. **History AI feedback (T6.5)**: Confirm whether this is in scope for this workstream or deferred. It requires a new API route or reuse of `/api/ai/lesson-chat` in a new mode.

4. **"Next question" skipping policy**: When a student force-advances past a question they scored below threshold on, should that be logged as a `'skipped'` attempt (with a partial score) so the SRS still schedules it for re-review? Current proposal is yes — store a skip attempt so the topic does not fall off the revision radar.
